# 交互式计划制定 Skill

## 目的

提供结构化的交互式方法来创建全面的实施计划。作为 Plan Mode 的超集/增强版，本 Skill 继承了 Plan Mode 的所有优势（并行探索、只读保护、增量计划、明确退出），同时融合了深度需求挖掘方法论（5层次框架、MoSCoW矩阵、边界情况矩阵）。

通过5阶段规划流程，引导用户从初始需求到详细的、可执行的计划，包含风险评估和技术权衡。

## 何时使用

在以下情况下触发此 skill：
- 用户需求不够明确
- 用户需要规划复杂任务的执行
- 需求在开始前需要澄清
- 存在多种解决方案需要评估
- 关键决策需要讨论
- 用户希望通过反馈循环迭代细化计划
- 任务涉及多个步骤或阶段
- 需要系统化地分析和拆解任务

## 核心原则

1. **只读模式** - 规划阶段只读取，不修改代码或配置（计划文件除外）
2. **增量计划** - 计划文件随阶段更新，作为进度追踪和沟通工具
3. **并行探索** - 复杂任务使用多 Agent 并行探索代码库
4. **用户控制** - 关键决策点询问用户，确保方向正确
5. **渐进式细化** - 从宽泛开始，迭代细化
6. **风险意识** - 尽早识别和缓解风险
7. **智能分组提问** - 相关问题分组提出（2-4个），依赖问题串行提问，避免信息过载
8. **多选题优先** - 尽可能提供选项而非开放式问题，降低用户回答成本
9. **增量验证** - 每个阶段完成后展示摘要（200-300字），验证理解正确性
10. **YAGNI 原则** - ruthlessly 删除非必要功能，保持方案简洁


### Phase 1: 深度理解（Initial Understanding）

**目标：** 全面理解用户需求和相关代码/资源

**⚠️ 重要：** Phase 1 的第一步是执行 1.1 并行探索，必须为每个维度启动 Explore Agent。

#### 1.1 并行探索（所有任务）

**⚠️ 强制要求：** 无论任务类型，都必须首先进行维度分析，然后为每个维度启动独立的 Explore Agent。这是 Phase 1 的第一步，不得跳过。

**执行流程：**

**第一步：维度分析**

根据任务类型，识别需要探索的维度。每个任务通常需要从 2-3 个维度进行探索：

**开发任务维度示例：**
- 维度1：现有实现模式（搜索相关功能的现有实现和架构）
- 维度2：相关组件和依赖（探索需要修改的组件及其依赖关系）
- 维度3：测试和验证模式（调查现有的测试模式和覆盖范围）

**写作任务维度示例：**
- 维度1：参考资料和风格（阅读相关文档和参考资料）
- 维度2：目标受众和结构（理解受众需求和内容结构）
- 维度3：约束和标准（了解字数、格式、规范要求）

**研究任务维度示例：**
- 维度1：背景资料（收集研究背景和相关文献）
- 维度2：研究方法（探索适用的研究方法和工具）
- 维度3：数据来源（识别关键信息源和数据渠道）

**业务流程任务维度示例：**
- 维度1：当前流程（理解现有业务流程和痛点）
- 维度2：相关方和约束（识别利益相关方和限制条件）
- 维度3：改进机会（分析优化空间和最佳实践）

**问题排查任务维度示例：**
- 维度1：问题现象（收集问题表现和相关信息）
- 维度2：系统背景（理解系统架构和运行环境）
- 维度3：可能原因（探索潜在原因和影响范围）

**第二步：启动 Explore Agent，并后台执行**

**关键要求：**
- ✅ 必须首先进行维度分析，识别 2-3 个探索维度
- ✅ 必须为每个维度启动独立的 Explore Agent
- ✅ 必须使用 Task tool 的 "Explore" subagent_type
- ✅ 必须在单个消息中并行启动所有 Agent（多个 Task tool 调用）
- ✅ 每个 Agent 的 prompt 必须具体明确，包含探索目标和方法
- ❌ 不得跳过维度分析步骤
- ❌ 不得将多个维度合并到一个 Agent 中
- ❌ 不得串行启动 Agent（必须并行）


**1. 后台执行参数**

启动 Explore agents 时，必须添加 `run_in_background: true` 参数：

```
Task tool:
- subagent_type: "Explore"
- description: "探索现有实现模式"
- prompt: "[探索任务详情]"
- run_in_background: true  # 关键：后台运行
```

**效果：**
- Agent 立即在后台启动，不阻塞主流程
- 可以立即启动下一个 Agent（实现真正并行）
- 主 Agent 继续执行后续步骤（如启动其他 Explore agents）

**2. 状态检查（非阻塞）**

在 Phase 1 的后续步骤中（如 1.2 深化探索、1.3 需求挖掘），可以使用 TaskOutput 工具非阻塞检查 agents 状态：

```
TaskOutput tool:
- task_id: [agent 的 task_id]
- block: false  # 非阻塞检查
```

**返回结果：**
- 如果 agent 已完成：返回探索结果
- 如果 agent 仍在运行：返回 "still running" 状态
- 如果 agent 失败：返回错误信息

**使用场景：**
- 在进行 1.3 需求挖掘时，顺便检查 agents 是否完成
- 如果已完成，可以立即使用探索结果
- 如果未完成，继续进行需求挖掘，稍后再检查

**3. 等待策略（阻塞）**

在 Phase 1 结束前（进入 1.5 更新计划文件之前），必须等待所有 Explore agents 完成：

```
TaskOutput tool:
- task_id: [agent 的 task_id]
- block: true  # 阻塞等待
- timeout: 600000  # 10分钟超时（可选）
```

**等待时机：**
- ✅ 在 1.5 更新计划文件之前，必须等待所有 agents 完成
- ✅ 在 Phase 2 方案设计之前，必须获取所有探索结果
- ❌ 不要在启动 agents 后立即等待（会失去并行优势）
- ❌ 不要在 1.3 需求挖掘时阻塞等待（可以并行进行）

**超时处理：**
- 如果 agent 超时或失败，记录错误信息
- 继续使用已完成的 agents 结果
- 在计划文件中标注探索不完整的维度


#### 1.2 深化探索（可选）

**说明：** 在 Explore Agent 完成初步探索后，如果需要更深入的理解，可以进行额外的信息收集和分析。

根据任务类型，采用不同的深化策略：

**开发任务：**
1. 深入阅读 Agent 识别的关键文件
2. 理解复杂的业务逻辑和算法
3. 分析性能瓶颈和优化机会
4. 研究相关的技术文档和 API

**写作任务：**
1. 深入研究 Agent 收集的参考资料
2. 分析优秀案例的写作技巧
3. 理解目标平台的内容偏好
4. 研究相关的 SEO 和传播策略

**研究任务：**
1. 深入阅读 Agent 识别的关键文献
2. 分析研究方法的适用性
3. 评估数据来源的可靠性
4. 研究相关的理论框架

**业务流程任务：**
1. 深入理解 Agent 识别的流程痛点
2. 分析相关方的利益和诉求
3. 研究行业最佳实践和标准
4. 评估改进方案的可行性

**问题排查任务：**
1. 深入分析 Agent 收集的问题信息
2. 理解系统的运行机制和依赖
3. 研究类似问题的解决方案
4. 评估修复方案的影响范围

**注意：** 此步骤是可选的，只在 Agent 探索结果不够充分时使用。

#### 1.3 需求深度挖掘（5层次框架）

**提问原则（关键）：**

1. **智能分组提问**
   - ✅ **相关问题分组**：同一主题的独立问题可以一次性提出（2-4个）
   - ✅ **依赖问题串行**：后续问题依赖前面回答时，分开提问
   - ❌ 不要：一次性列出 10+ 个问题让用户疲于应对

2. **多选题优先**
   - ❌ 不要："你希望实现什么功能？"（开放式）
   - ✅ 应该："你希望实现哪些功能？A. 用户登录 B. 用户注册 C. 密码重置 D. 其他"

3. **智能跳过**
   - 如果用户已经说明，不要重复问
   - 如果可以从常识推断，不要问
   - 只问真正不明确的关键点

**提问策略示例：**

✅ **好的分组（第1轮 - 目标与范围）：**
```
关于这个任务，我需要了解几个关键点：

1. **任务目标**：这个任务要解决什么问题？
   A. 提升用户体验
   B. 修复现有 bug
   C. 添加新功能
   D. 性能优化
   E. 其他（请说明）

2. **核心功能**：必须包含哪些功能？（可多选）
   A. 功能1
   B. 功能2
   C. 功能3
   D. 其他（请说明）

3. **时间约束**：是否有截止日期？
   A. 本周内
   B. 本月内
   C. 无明确期限
```

等待回答后，根据回答决定下一轮问题。

✅ **好的串行（依赖前面回答）：**
```
[用户回答了"添加用户认证功能"]

了解了，那认证方式是？
A. 用户名+密码（推荐，简单）
B. 手机号+验证码
C. OAuth（Google/GitHub）
D. 其他（请说明）

[等待回答后，再问下一个依赖问题]
```

**提问流程：**
按照 5 层次框架分组提问，通常 2-3 轮完成：
- **第1轮**：目标（Why）+ 核心范围（What）→ 2-4 个独立问题
- **第2轮**：执行方式（How）+ 约束（Constraints）→ 2-3 个独立问题
- **第3轮**：特殊情况（Special Cases）+ 澄清细节 → 1-2 个问题

总计约 5-9 个问题，分 2-3 轮进行。

---

**提问前的智能分析：**
1. 仔细分析用户的初始描述，识别已明确的信息
2. 识别可从上下文推断的隐含信息
3. 只针对真正不明确的关键点提问

使用**5层次框架**系统化地深度澄清需求：

**第1层：目标与价值（Why）**
- 为什么需要完成这个任务？解决什么问题？
- 这个任务的成功标准是什么？如何衡量？
- 对相关方的价值是什么？

**第2层：范围与内容（What）**
- 核心内容/功能是什么？
- 哪些是必须完成的（Must Have）？
- 哪些是应该完成的（Should Have）？
- 哪些是可以完成的（Could Have）？
- 明确不做的内容有哪些（Won't Have）？

**第3层：执行方式（How）**
- 用户如何触发这个功能？
- 成功/失败时用户应该看到什么？
- 是否需要加载状态、进度提示、错误提示？
- 是否需要确认对话框或撤销功能？
- 如何开始这个任务？
- 关键步骤是什么？
- 需要哪些资源和工具？
- 如何呈现最终成果？

**第4层：约束与限制（Constraints）**
- 性能要求：响应时间？并发量？数据量？
- 安全要求：涉及敏感数据？需要加密？
- 兼容性：支持哪些浏览器/设备？
- 依赖限制：必须使用特定库？
- 时间限制：截止日期？时间预算？
- 资源限制：预算？人力？工具？
- 质量要求：标准？规范？
- 其他约束：政策？法规？偏好？

**第5层：特殊情况（Special Cases）**
- 网络请求失败怎么办？
- 数据不存在怎么办？
- 用户权限不足怎么办？
- 并发冲突怎么办？
- 用户输入无效数据怎么办？
- 如果遇到意外情况怎么办？
- 如果资源不足怎么办？
- 如果时间不够怎么办？
- 如果结果不理想怎么办？

**智能提问规则：**
- ✅ 如果用户说"添加用户认证"，推断需要登录/注册/密码重置，不再问"需要哪些功能"
- ✅ 如果用户说"实现深色模式"，推断需要主题切换，不再问"用户如何触发"
- ✅ 如果用户说"写一本书"，推断需要大纲、章节、内容，不再问"需要哪些部分"
- ✅ 如果用户说"做市场调研"，推断需要数据收集、分析、报告，不再问"需要哪些步骤"
- ❌ 不要问用户已经明确说明的内容
- ❌ 不要问可以从常识推断的内容
- ❌ 不要问与功能无关的通用问题

#### 1.4 澄清歧义

使用 AskUserQuestion 工具，**智能分组提问**：
- 相关的独立问题可以一次性提出（2-4个）
- 依赖前面回答的问题分开提问

**提问技巧：**
- 优先使用多选题，提供 2-4 个选项
- 每个选项都包含简短说明
- 将推荐选项放在第一位，标注"（推荐）"
- 始终提供"其他"选项让用户自由输入

**示例：**

❌ **不好的提问方式（一次性太多）：**
```
请回答以下问题：
1. 目标用户是谁？
2. 主要功能有哪些？
3. 性能要求是什么？
4. 安全要求是什么？
5. 兼容性要求是什么？
6. 时间限制是什么？
7. 预算限制是什么？
8. 质量标准是什么？
```

✅ **好的提问方式（第1轮 - 分组独立问题）：**
```
关于这个任务，我需要了解几个关键点：

1. **目标用户**：
   A. 内部员工（推荐，简化权限管理）
   B. 外部客户
   C. 合作伙伴
   D. 其他（请说明）

2. **核心功能**（可多选）：
   A. 功能1
   B. 功能2
   C. 功能3
   D. 其他（请说明）

3. **时间约束**：
   A. 本周内
   B. 本月内
   C. 无明确期限
```

等待回答后，根据回答决定第2轮问题（如性能、安全等）。

**检查点：** 确认对需求的理解是否正确。

#### 1.4.5 展示理解摘要

在更新计划文件之前，先向用户展示理解摘要（200-300字）：

```
📋 **Phase 1 理解摘要**

**任务目标：**
[一句话描述]

**核心需求：**
- [需求1]
- [需求2]
- [需求3]

**关键约束：**
- [约束1]
- [约束2]

**探索发现：**
- [发现1]
- [发现2]

这个理解是否正确？是否需要调整？
```

等待用户确认后，再进行 1.5 更新计划文件。

#### 1.5 更新计划文件

将探索发现和需求理解写入计划文件（`docs/spec/[功能名称]-[YYYYMMDD-HHmm].md`）：
- 任务概述
- 探索发现（关键文件、现有模式、依赖关系）
- 需求理解（5层次框架分析结果）

### Phase 2: 方案设计（Design）

**目标：** 设计实现方案，明确范围和边界

#### 2.1 方案设计

**⚠️ 强制要求：** 必须启动 Plan Agent 进行深度方案设计。

**解决方案设计**（如有多个方案）：
- 候选解决方案对比
- 各方案的优缺点分析
- 效率/质量/可行性权衡
- 推荐方案和理由

**Plan Agent 调用示例：**

```
Task tool:
- subagent_type: "Plan"
- description: "设计[功能名称]实现方案"
- prompt: [计划任务详情]
```

#### 2.2 范围确认（MoSCoW 矩阵）

**⚠️ YAGNI 原则（关键）：**

在划分范围时，必须 **ruthlessly（无情地）删除非必要功能**：

- ❌ 不要添加"可能有用"的功能
- ❌ 不要添加"未来可能需要"的功能
- ❌ 不要添加"锦上添花"的功能
- ✅ 只保留"现在必须"的功能

**判断标准：**
- 如果去掉这个功能，核心价值是否受影响？
  - 是 → Must Have
  - 否 → 删除或降级到 Won't Have

**示例：**

❌ **过度设计：**
```
Must Have:
- 用户登录
- 用户注册
- 密码重置
- OAuth 集成
- 双因素认证
- 生物识别
```

✅ **YAGNI 后：**
```
Must Have:
- 用户登录
- 用户注册

Won't Have（现在不做）:
- 密码重置（可以手动重置）
- OAuth 集成（未来再说）
- 双因素认证（未来再说）
- 生物识别（未来再说）
```

使用 **MoSCoW 矩阵**明确范围：

```markdown
**Must Have（必须完成）**
- [ ] 内容1 - 理由：核心价值
- [ ] 内容2 - 理由：关键要求

**Should Have（应该完成）**
- [ ] 内容3 - 理由：重要但非关键

**Could Have（可以完成）**
- [ ] 内容4 - 理由：锦上添花

**Won't Have（不完成）**
- [ ] 内容5 - 理由：超出范围
```

**检查点：** 使用 AskUserQuestion 确认用户同意此范围划分。

#### 2.3 边界情况分析

根据任务类型，选择合适的边界情况矩阵进行系统化分析。

**矩阵1：技术+业务矩阵**（用于开发任务）

| 维度 | 典型场景 | 处理方向 |
|------|---------|---------|
| **数据** | 空/大/错/缺/重复 | 验证+降级+默认值 |
| **网络** | 超时/断开/错误/慢 | 重试+缓存+提示 |
| **并发** | 重复提交/冲突/竞态 | 防抖/锁/序列号 |
| **权限** | 未登录/无权限/过期 | 跳转/刷新/提示 |
| **业务规则** | 规则冲突/特殊日期/历史数据/地区差异 | 优先级/配置化/兼容 |
| **特殊用户** | 新用户/高级用户/残障用户/移动用户 | 引导/快捷/无障碍/响应式 |

**矩阵2：项目管理矩阵**（用于通用任务）

| 维度 | 典型场景 | 处理方向 |
|------|---------|---------|
| **资源** | 预算不足/人力不足/工具缺失 | 调整范围/寻求支持 |
| **时间** | 进度延迟/时间冲突/截止日期提前 | 调整计划/重排优先级 |
| **质量** | 成果不达标/标准变更/技术债务 | 返工/评估影响/规划偿还 |
| **依赖** | 信息缺失/依赖延迟/API变更 | 替代方案/并行处理 |
| **外部** | 需求变更/外部干扰/政策变化 | 评估影响/重新规划 |
| **意外** | 突发情况/不可抗力/安全事件 | 应急预案/退出策略 |

**使用指南：**
- 开发任务（Phase 1 并行探索代码库）→ 使用技术+业务矩阵
- 其他任务（写作/研究/规划/排查）→ 使用项目管理矩阵
- 详细的场景、触发条件、处理方案和测试用例，参见 `references/planning-best-practices.md`

### Phase 3: 审查对齐（Review）

**目标：** 验证方案，确保与用户意图一致

#### 3.1 阅读关键文件

深入阅读 Phase 1 和 Phase 2 中识别的关键文件，验证方案的可行性。

#### 3.2 风险评估

- 风险识别和缓解措施
- 依赖风险识别和缓解措施
- 质量/时间风险识别和缓解措施

#### 3.3 验证方法

- 验证标准：质量目标、验收标准
- 检验方式：关键检查点、验证流程
- 反馈机制：如何收集反馈、如何改进

#### 3.4 完整性检查

- [ ] 所有需求都有对应执行步骤
- [ ] 所有风险都有缓解措施
- [ ] 所有特殊情况都有处理方案
- [ ] 验证方法覆盖所有关键环节
- [ ] 执行顺序合理，无循环依赖

#### 3.5 澄清剩余问题

**提问原则：**

- 相关问题可以分组提出（2-3个）
- 依赖问题分开提问

**检查点：** 确保计划与用户意图完全一致。

### Phase 4: 最终计划（Final Plan）

**目标：** 整理最终计划文件

#### 4.1 整理计划文件

将计划文件整理为最终版本，结构如下：

```markdown
# [任务名称] 执行计划

**状态：** 已批准

---

## 需求

### 目标
[一句话描述要达成的目标]

### 范围

**必须完成：**
- [ ] [功能1]
- [ ] [功能2]

**不在范围内：**
- [明确排除的内容]

### 验收标准
- [ ] [可验证的标准1]
- [ ] [可验证的标准2]

---

## 方案

### 概述
[一段话描述技术方案]

### 执行步骤

#### 步骤 1：[步骤名称]

**任务 1.1：[任务描述]**
- 文件：`path/to/file.ts`
- 操作：[新增/修改/删除]
- 详情：
  ```typescript
  // 示例代码或伪代码
  ```

**任务 1.2：[任务描述]**
- 文件：`path/to/file.ts`
- 操作：[新增/修改/删除]
- 详情：[具体说明]

#### 步骤 2：[步骤名称]

**任务 2.1：[任务描述]**
- 文件：`path/to/file.ts`
- 操作：[新增/修改/删除]
- 详情：[具体说明]

### 步骤依赖

| 步骤 | 依赖 | 说明 |
|------|------|------|
| 步骤 1 | - | 无依赖，可立即执行 |
| 步骤 2 | - | 无依赖，可与步骤 1 并行 |
| 步骤 3 | 步骤 1, 步骤 2 | 需等步骤 1 和步骤 2 完成 |

### 注意事项
- [关键约束或风险点]
- [需要特别注意的边界情况]
```

#### 4.2 最终计划原则

- **聚焦需求和方案**，去除过程性内容
- **任务详情充分**，包含文件路径、操作类型、代码示例
- **便于用户审核**，快速判断需求和方案是否符合预期
- **便于AI执行**，对照文档即可准确实现需求
- **步骤依赖必须填写**，根据任务实际依赖关系填写"步骤依赖"章节，`-` 表示无依赖可并行，步骤名表示必须等待前置步骤完成
