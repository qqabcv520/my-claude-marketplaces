# 交互式计划制定 Skill

## 目的

提供结构化的交互式方法来创建全面的实施计划。作为 Plan Mode 的超集/增强版，本 Skill 继承了 Plan Mode 的所有优势（并行探索、只读保护、增量计划、明确退出），同时融合了深度需求挖掘方法论（5维度功能需求挖掘、递归式功能树细化、技术性验收标准）。

通过5阶段规划流程，引导用户从初始需求到详细的、可执行的计划，包含风险评估和技术权衡。

## 何时使用

在以下情况下触发此 skill：
- 用户需求不够明确
- 用户需要规划复杂任务的执行
- 需求在开始前需要澄清
- 存在多种解决方案需要评估
- 关键决策需要讨论
- 用户希望通过反馈循环迭代细化计划
- 任务涉及多个步骤或阶段
- 需要系统化地分析和拆解任务

## 核心原则

1. **只读模式** - 规划阶段只读取，不修改代码或配置（计划文件除外）
2. **增量计划** - 计划文件随阶段更新，作为进度追踪和沟通工具
3. **并行探索** - 复杂任务使用多 Agent 并行探索代码库
4. **用户控制** - 关键决策点询问用户，确保方向正确
5. **渐进式细化** - 从宽泛开始，迭代细化
6. **风险意识** - 尽早识别和缓解风险
7. **智能分组提问** - 相关问题分组提出（2-4个），依赖问题串行提问，避免信息过载
8. **多选题优先** - 尽可能提供选项而非开放式问题，降低用户回答成本
9. **强制使用 AskUserQuestion 工具** - 所有向用户提问必须使用 AskUserQuestion 工具，禁止在普通文本中提问
10. **增量验证** - 每个阶段完成后展示摘要（200-300字），验证理解正确性
11. **YAGNI 原则** - ruthlessly 删除非必要功能，保持方案简洁
12. **零上下文假设** - 最终计划必须假设执行者对代码库零了解。所有任务描述必须自包含：精确文件路径、完整代码片段、可运行命令及预期输出，使任何人（包括 AI Agent）对照计划即可准确执行


### Phase 1: 深度理解（Initial Understanding）

**目标：** 全面理解用户需求和相关代码/资源

**Phase 1 的第一步是执行 1.1 并行探索，必须为每个维度启动 Explore Agent。**

#### 1.1 并行探索（所有任务）

**强制要求：** 无论任务类型，都必须首先进行维度分析，然后为每个维度启动独立的 Explore Agent。这是 Phase 1 的第一步，不得跳过。

**执行流程：**

**第一步：维度分析**

根据任务类型，识别需要探索的维度。每个任务通常需要从 2-3 个维度进行探索：

**开发任务维度示例：**
- 维度1：现有实现模式（搜索相关功能的现有实现和架构）
- 维度2：相关组件和依赖（探索需要修改的组件及其依赖关系）
- 维度3：测试和验证模式（调查现有的测试模式和覆盖范围）

**写作任务维度示例：**
- 维度1：参考资料和风格（阅读相关文档和参考资料）
- 维度2：目标受众和结构（理解受众需求和内容结构）
- 维度3：约束和标准（了解字数、格式、规范要求）

**研究任务维度示例：**
- 维度1：背景资料（收集研究背景和相关文献）
- 维度2：研究方法（探索适用的研究方法和工具）
- 维度3：数据来源（识别关键信息源和数据渠道）

**业务流程任务维度示例：**
- 维度1：当前流程（理解现有业务流程和痛点）
- 维度2：相关方和约束（识别利益相关方和限制条件）
- 维度3：改进机会（分析优化空间和最佳实践）

**问题排查任务维度示例：**
- 维度1：问题现象（收集问题表现和相关信息）
- 维度2：系统背景（理解系统架构和运行环境）
- 维度3：可能原因（探索潜在原因和影响范围）

**第二步：启动 Explore Agent，并后台执行**

**关键要求：**
- 必须首先进行维度分析，识别 2-3 个探索维度
- 必须为每个维度启动独立的 Explore Agent
- 必须使用 Task tool 的 "Explore" subagent_type
- 必须在单个消息中并行启动所有 Agent（多个 Task tool 调用）
- 每个 Agent 的 prompt 必须具体明确，包含探索目标和方法
- 不得跳过维度分析步骤
- 不得将多个维度合并到一个 Agent 中
- 不得串行启动 Agent（必须并行）


**并行启动与等待**

在**单个消息**中同时调用多个 Agent tool，它们会自动并行执行，主 Agent 阻塞等待所有结果返回后再继续：

```
# 在同一条消息中并行启动所有 Agent（不使用 run_in_background）
Task tool 1:
- subagent_type: "Explore"
- description: "探索维度1"
- prompt: "[维度1探索任务详情]"

Task tool 2:
- subagent_type: "Explore"
- description: "探索维度2"
- prompt: "[维度2探索任务详情]"
```

**执行策略：**
- 所有 Explore agents 在单条消息中并行启动，主 Agent 自动等待全部完成
- **不要**使用 `run_in_background: true`（那会让主 Agent 跳过等待继续执行）
- 收到所有 agents 的结果后，再继续进入 1.2 及后续步骤

**失败处理：**
- 如果某个 agent 失败，记录错误信息，继续使用已完成的 agents 结果
- 在计划文件中标注探索不完整的维度


#### 1.2 深化探索（可选）

**说明：** 在 Explore Agent 完成初步探索后，如果需要更深入的理解，可以进行额外的信息收集和分析。

根据任务类型，采用不同的深化策略：

**开发任务：**
1. 深入阅读 Agent 识别的关键文件
2. 理解复杂的业务逻辑和算法
3. 分析性能瓶颈和优化机会
4. 研究相关的技术文档和 API

**写作任务：**
1. 深入研究 Agent 收集的参考资料
2. 分析优秀案例的写作技巧
3. 理解目标平台的内容偏好
4. 研究相关的 SEO 和传播策略

**研究任务：**
1. 深入阅读 Agent 识别的关键文献
2. 分析研究方法的适用性
3. 评估数据来源的可靠性
4. 研究相关的理论框架

**业务流程任务：**
1. 深入理解 Agent 识别的流程痛点
2. 分析相关方的利益和诉求
3. 研究行业最佳实践和标准
4. 评估改进方案的可行性

**问题排查任务：**
1. 深入分析 Agent 收集的问题信息
2. 理解系统的运行机制和依赖
3. 研究类似问题的解决方案
4. 评估修复方案的影响范围

**注意：** 此步骤是可选的，只在 Agent 探索结果不够充分时使用。

#### 1.3 WBS工作分解（中等/复杂/系统级任务必须执行）

**触发条件：** 当任务属于"中等"、"复杂"或"系统级"复杂度时（见复杂度适配表），必须执行此步骤。仅简单任务跳过。

**目标：** 通过三阶段循环提问机制，将复杂任务从粗粒度骨架逐步细化为可执行的完整功能树。

**叶节点标准（量化检查清单）：**

一个 Story 达到叶节点需满足以下所有条件：

✓ **工作量**：2-4小时可完成
✓ **文件范围**：涉及文件数 ≤ 3个
✓ **代码规模**：新增/修改代码 ≤ 200行
✓ **测试规模**：测试用例 ≤ 5个
✓ **依赖关系**：依赖的外部接口 ≤ 2个
✓ **决策点**：无需用户进一步决策
✓ **可描述性**：可以用一句话清晰描述

**注意：** 这些是指导性标准，不是硬性限制。如果某个 Story 略微超出某项指标但整体合理，可以保留。关键是确保 Story 足够小、独立、可执行。

**层次结构：** Epic（一级模块）→ Feature（二级功能）→ Story（具体行为，叶节点）

### 阶段一：搭建骨架

**目标：** 快速搭出系统骨架，识别模块边界，不追求叶节点完整。

**第一步：识别一级模块**

使用"用户旅程法"：沿着用户使用系统的完整路径，逐步识别需要的模块。

**示例（电商系统）：**
```
用户旅程：注册 → 浏览商品 → 加入购物车 → 下单 → 支付 → 查看订单 → 评价
→ 识别一级模块：用户管理、商品管理、购物车、订单管理、支付、评价
→ 补充支撑模块：权限管理、通知系统、后台管理
```

**第二步：二级功能粗略列出**

对每个一级模块，使用 "CRUD+" 法（增删改查 + 业务特有操作）粗略列出二级功能。

**第三步：标记确定性级别**

对每个模块，标记确定性级别：
- **确定**：需求明确，可直接规划
- **待澄清**：存在多种实现方式，需与用户确认
- **待定义**：用户可能未考虑到，需要主动提出

**第四步：展示并确认**

向用户展示粗粒度分解树，重点讨论：
1. 模块边界是否正确？
2. 待澄清的模块如何决策？
3. 待定义的模块是否需要？
4. 是否有遗漏的支撑模块（权限、日志、配置、监控等）？

---

### 阶段二：循环钻取

**目标：** 通过递归式提问，将每个模块从粗粒度逐步钻取到叶节点。

**钻取原则：**

1. **逐层钻取，不跨层**
   - 先确认"要不要"（Feature 级），再问"怎么要"（Story 级）
   - 用户确认某功能存在后，才钻取该功能的子细节
   - 用户否定某功能后，跳过该分支的所有子问题

2. **同层并行，跨层串行**
   - 同一模块下的同层子功能可以一次性提出（3-6个）
   - 下一层的问题必须等上一层回答后才能提出

**提问原则（关键）：**

**⚠️ 强制要求：所有向用户提问必须使用 AskUserQuestion 工具，禁止在普通文本中提问。**

1. **智能分组提问**
   - 相关问题分组：同一主题的独立问题可以一次性提出（2-4个）
   - 依赖问题串行：后续问题依赖前面回答时，分开提问
   - 不要：一次性列出 10+ 个问题让用户疲于应对

2. **多选题优先**
   - 不要："你希望实现什么功能？"（开放式）
   - 应该："你希望实现哪些功能？A. 用户登录 B. 用户注册 C. 密码重置 D. 其他"

3. **智能跳过**
   - 如果用户已经说明，不要重复问
   - 如果可以从常识推断，不要问
   - 只问真正不明确的关键点

**5维度辅助思考框架：**

在每层钻取时，用以下维度辅助思考要问哪些子问题（不直接暴露给用户）：

**维度1：目标与背景**
- 解决什么问题？业务目标是什么？
- 怎么算成功？（量化成功指标）

**维度2：用户与角色**
- 谁来用？有几种角色？
- 各角色的核心诉求？权限模型？

**维度3：核心流程与交互**
- 用户的完整操作路径是什么？（从入口到完成）
- 关键状态如何流转？（如：草稿→审核→发布→下架）
- 有哪些分支路径？（如：取消、退回、跳过）
- 各步骤之间的触发条件是什么？

**维度4：数据与内容**
- 涉及哪些核心数据实体？各实体包含哪些关键信息？
- 数据之间的关系是什么？（一对多、多对多）
- 数据的生命周期？（创建→更新→归档→删除）
- 需要展示哪些内容？列表/详情/统计各需要什么字段？

**维度5：边界与规则**
- 核心业务规则是什么？（如：库存不足不可下单、同一用户每天限3次）
- 哪些功能在范围内？哪些明确排除？
- 异常场景如何处理？（如：支付超时、数据冲突、并发操作）
- 有哪些数量/时间/权限上的限制？

**钻取流程示例：**

```
第1轮（Feature 级）：对每个 Epic 列出可能的 Feature，让用户勾选

  你选择了【评论系统】，需要包含哪些功能？
  A. 富文本编辑（Markdown）
  B. 登录后评论
  C. 嵌套回复（评论的评论）
  D. 评论审核机制
  E. 其他：___

  你选择了【用户认证】，需要包含哪些功能？
  A. 邮箱密码登录
  B. GitHub OAuth
  C. Google OAuth
  D. 用户个人主页
  E. 其他：___
```

```
第2轮（Story 级）：对用户选中的 Feature，继续钻取子细节

  你确认评论支持【Markdown】，继续确认：
  1. Markdown 是否支持嵌入 HTML 标签？
  2. 是否需要实时预览？
  3. 是否支持图片上传？

  你确认需要【评论审核】，继续确认：
  1. 审核模式？A. 先审后发 B. 先发后审 C. 仅首次评论需审核
```

```
第3轮+（按需继续）：直到所有分支达到叶节点

  你确认 Markdown 支持【图片上传】，继续确认：
  1. 图片存储方式？A. 本地存储 B. 对象存储（S3等）C. 外链引用
  2. 单张图片大小限制？A. 2MB B. 5MB C. 10MB D. 自定义
```

**终止条件：**
- 所有选中的功能分支都已钻取到叶节点
- 叶节点符合量化标准（见 1.3 叶节点标准）：工作量 2-8小时、文件数 ≤3、代码 ≤200行、测试 ≤5个、依赖 ≤2个、无需用户决策、可一句话描述

**继续钻取的信号：**
- Story 描述模糊，无法直接编写代码
- Story 涉及多个独立的子任务
- Story 工作量超过 6 小时
- Story 需要用户进一步决策（如选择技术方案、确定业务规则）

**智能提问规则：**
- 如果用户说"添加用户认证"，推断需要登录/注册/密码重置，不再问"需要哪些功能"
- 如果用户说"实现深色模式"，推断需要主题切换，不再问"用户如何触发"
- 不要问用户已经明确说明的内容
- 不要问可以从常识推断的内容

### 阶段三：最终确认

**目标：** 验证完整性，应用 YAGNI 原则，确保功能树可执行。

**YAGNI 检查：**
- 逐个检查每个功能：如果去掉这个功能，核心价值是否受影响？
- 否 → 标记为"可选"或直接删除
- 是 → 保留

**完整性检查：**
1. 是否有遗漏的模块？
2. 是否有需要删除的非必要功能？
3. 待澄清/待定义的模块如何决策？
4. 是否有遗漏的支撑模块（权限、日志、配置、监控等）？

**确定性标记：**
- **确定**：需求明确，可直接规划
- **待澄清**：存在多种实现方式，需与用户确认
- **待定义**：用户可能未考虑到，需要主动提出

**展示给用户确认：**
展示完整的功能分解树，包含所有 Epic、Feature、Story，以及确定性标记，获取用户最终确认。

#### 1.4 用户故事级验收条件（复杂/系统级任务必须执行）

**目标：** 从用户视角定义"什么算完成"，不涉及技术细节。

**格式示例：**
- "用户能通过邮箱注册账号并收到验证邮件"
- "管理员能查看所有用户的订单列表，支持按日期筛选"
- "支付失败时用户能看到明确的失败原因并重试"

**规则：**
- 异常路径也用用户语言描述（"网络断开时用户能看到提示"），不写技术实现
- 覆盖所有核心 Story，不限于 Must
- 不写 Given/When/Then，不涉及具体技术实现（技术性 AC 在 Phase 2 完成）

---

#### 1.5 澄清歧义

使用 AskUserQuestion 工具，**智能分组提问**：
- 相关的独立问题可以一次性提出（2-4个）
- 依赖前面回答的问题分开提问
- 包括分解和验收条件中发现的新问题

**提问技巧：**
- 优先使用多选题，提供 2-4 个选项
- 每个选项都包含简短说明
- 将推荐选项放在第一位，标注"（推荐）"
- 始终提供"其他"选项让用户自由输入

**检查点：** 确认对需求的理解是否正确。

#### 1.5.5 展示理解摘要

在更新计划文件之前，先向用户展示理解摘要（200-300字）：

```
**Phase 1 理解摘要**

**任务目标：**
[一句话描述]

**核心需求：**
- [需求1]
- [需求2]
- [需求3]

**关键约束：**
- [约束1]
- [约束2]

**探索发现：**
- [发现1]
- [发现2]

这个理解是否正确？是否需要调整？
```

等待用户确认后，再进行 1.6 更新计划文件。

#### 1.6 更新计划文件

将探索发现和需求理解写入计划文件（`docs/spec/[功能名称]-[YYYYMMDD-HHmm].md`）：
- 任务概述
- 探索发现（关键文件、现有模式、依赖关系）
- 需求理解（5维度功能需求分析结果）
- 分解树骨架（复杂/系统级）
- 用户故事级验收条件（复杂/系统级）

### Phase 2: 方案设计（Design）

**目标：** 设计实现方案，明确技术规约和验收标准

#### 2.1 技术方案设计

**强制要求：** 必须启动 Plan Agent 进行深度方案设计。

**技术约束收集：**

在设计方案前，先明确技术约束（Phase 1 聚焦功能需求，技术约束在此阶段收集）：

- **质量属性**：性能（响应时间/并发量/数据量）、安全（认证/加密/审计）、可用性、可扩展性
- **约束条件**：技术栈、预算、时间期限、团队规模、法规合规
- **集成环境**：已有系统/API/数据库、第三方服务、可复用资源

如果 Phase 1 探索已覆盖部分技术约束，直接复用，不重复提问。

**解决方案设计**（如有多个方案）：
- 候选解决方案对比
- 各方案的优缺点分析
- 效率/质量/可行性权衡
- 推荐方案和理由

**多方案时使用方案对比框架（6维度）：**

| 维度 | 评估标准 |
|------|---------|
| **实现复杂度** | 代码量、学习曲线、维护成本 |
| **性能** | 响应时间、资源消耗、可扩展性 |
| **可维护性** | 代码清晰度、文档完整性、社区支持 |
| **兼容性** | 浏览器支持、依赖版本、向后兼容 |
| **安全性** | 已知漏洞、安全最佳实践、审计记录 |
| **成本** | 开发时间、第三方服务费用、运维成本 |


#### 2.2 模块规约细化（复杂/系统级任务必须执行）

**触发条件：** 当任务属于"复杂"或"系统级"复杂度时，必须执行此步骤。

**目标：** 为功能分解树中的每个模块产出完整的**技术契约**。

**执行流程：**

**第一步：逐模块产出规约**

对功能分解树中的每个一级模块，使用 Plan Agent **并行**产出以下规约：

```
Task tool:
- subagent_type: "Plan"
- description: "细化[模块名]规约"
- prompt: |
    基于以下功能分解和需求上下文，为[模块名]产出详细规约：
    [粘贴功能分解树和需求上下文]

    需要产出：
    1. 数据模型（字段名、类型、约束、关联关系）
    2. API/接口契约（路径、方法、入参、出参、状态码、错误码）
    3. 业务规则（所有条件分支用 if/then/else 明确表达，量化边界值）
    4. 页面/组件规约（如涉及前端：组件名、props、state、交互行为）
    5. 关键决策记录（选了什么方案、为什么）
- run_in_background: true
```

**关键要求：**
- 每个一级模块启动独立的 Plan Agent（并行）
- 规约必须具体到字段级别、参数级别
- 所有业务规则必须用明确的条件表达（if/then/else），不能用模糊描述
- 禁止出现"根据情况"、"合理处理"、"适当的"等模糊词汇

**第二步：展示规约摘要并确认**

向用户展示每个模块的关键规约，重点确认：
1. 数据模型是否正确
2. 业务规则是否符合预期
3. 接口设计是否合理

#### 2.3 技术性验收标准（复杂/系统级任务必须执行）

**触发条件：** 复杂/系统级任务必须执行。

**前置条件：** 2.2 模块规约已完成（此时有数据模型、API 契约、业务规则可引用）。

**格式：** Given [前置条件] / When [操作] / Then [预期结果]

**必须覆盖：** 正常路径 + 异常路径

**异常路径强制检查维度**（写 AC 时的思考清单）：
- 数据异常（空/大/错/缺）
- 网络异常（超时/断开）
- 权限异常（未登录/无权限/过期）
- 并发异常（重复提交/冲突）

**量化下限：** 每个核心 Story 至少覆盖上述 4 个维度中的 2 个异常路径，防止走过场。

**执行方式：** 为每个一级模块启动独立 Plan Agent 并行编写。

展示给用户确认。

#### 2.4 范围再确认

**时机：** 技术方案和规约完成后，此时对工作量有真实认知。

**目标：** 用工作量信息校准范围，避免"看似简单实则巨大"的功能。

**执行方式：**
1. 列出所有 Story 的预估工作量
2. 标记工作量/价值比不合理的 Story
3. 向用户展示并确认：哪些保留、哪些删除/降级/简化

**YAGNI 原则再次应用：** 如果去掉这个功能，核心价值是否受影响？否 → 删除。

**检查点：** 使用 AskUserQuestion 确认用户同意范围调整。

#### 2.4.5 展示方案摘要

在更新计划文件之前，先向用户展示方案摘要（200-300字）：

```
**Phase 2 方案摘要**

**推荐方案：**
[一句话描述技术方案]

**核心步骤：**
1. [步骤1]
2. [步骤2]
3. [步骤3]

**范围确认：**
- 保留：[功能列表]
- 删除/降级：[功能] - 理由：[...]

**关键风险：**
- [风险1] - 缓解措施：[...]

这个方案是否合理？是否需要调整？
```

等待用户确认后，再进行 2.5 更新计划文件。

#### 2.5 更新计划文件

追加方案设计内容到计划文件：
- 模块规约（含业务规则）
- 技术性验收标准

- 执行步骤

### Phase 3: 审查对齐（Review）

**目标：** 验证方案，确保与用户意图一致

#### 3.1 阅读关键文件

深入阅读 Phase 1 和 Phase 2 中识别的关键文件，验证方案的可行性。

#### 3.2 风险评估

**风险识别矩阵（6类）：**

| 风险类型 | 识别问题 | 示例 |
|---------|---------|------|
| **技术风险** | 技术是否成熟？是否有经验？ | 使用新框架、复杂算法 |
| **依赖风险** | 是否依赖外部服务或库？ | 第三方 API、未维护的库 |
| **性能风险** | 是否可能影响性能？ | 大数据处理、复杂查询 |
| **安全风险** | 是否涉及敏感数据？ | 用户认证、支付信息 |
| **兼容性风险** | 是否需要支持多平台/浏览器？ | 跨浏览器、移动端适配 |
| **时间风险** | 任务是否可能超时？ | 复杂重构、大规模迁移 |

**评估模板：**

```markdown
**风险：** [风险描述]
**概率：** 高/中/低
**影响：** 高/中/低
**优先级：** 概率 x 影响
**缓解措施：**
- 预防措施：[如何降低概率]
- 应对措施：[如果发生如何处理]
- 备选方案：[Plan B]
```

**优先级矩阵：**

```
影响
高 | 中优先级 | 高优先级 | 高优先级
中 | 低优先级 | 中优先级 | 高优先级
低 | 低优先级 | 低优先级 | 中优先级
     低        中        高     → 概率
```

#### 3.3 完整性与确定性检查

合并所有检查为一次，避免重复：

**完整性：**
- [ ] 所有需求都有对应执行步骤
- [ ] 所有风险都有缓解措施

**可执行性：**
- [ ] 每个任务具体可操作
- [ ] 文件路径真实
- [ ] 依赖关系可行

**确定性（换人测试）：**
- [ ] 数据模型完整 — 每个实体的字段名、类型、约束、关联关系都已定义
- [ ] 接口契约完整 — 每个 API 的路径、方法、入参、出参、错误码都已定义
- [ ] 业务规则无歧义 — 检查 2.2 规约中的业务规则是否用 if/then/else 明确表达
- [ ] 技术选型已锁定 — 所有库、框架、工具版本已确定
- [ ] 边界值已量化 — 分页大小、超时时间、重试次数、字段长度限制等都是具体数字

**模糊词检查：** 禁止"适当的"、"合理的"、"根据情况"等模糊词。

**测试策略要点：**
- [ ] 计划中包含测试层次（单元/集成/E2E）
- [ ] 关键路径覆盖明确（"测什么"和"怎么测"的核心要点）

**发现不确定项的处理：**
1. 如果可以通过探索代码或查阅文档确定 → 立即确定并更新计划
2. 如果需要用户决策 → 在 3.4 中向用户提问，提供推荐选项
3. 如果属于实现细节且有行业惯例 → 直接采用惯例并在计划中注明

#### 3.4 澄清剩余问题

使用 AskUserQuestion **智能分组提问**，澄清任何剩余的歧义或不确定性。

**提问原则：**
- 优先使用多选题
- 相关问题可以分组提出（2-3个）
- 依赖问题分开提问

**检查点：** 确保计划与用户意图完全一致。

#### 3.4.5 展示审查摘要

在更新计划文件之前，先向用户展示审查摘要（200-300字）：

```
**Phase 3 审查摘要**

**风险评估：**
- [风险1] - 缓解措施：[...]
- [风险2] - 缓解措施：[...]

**测试策略要点：**
- [测试层次和覆盖范围]

**完整性检查：**
- 所有需求都有对应执行步骤
- 所有风险都有缓解措施

**调整说明：**
- [调整1]
- [调整2]

这个审查结果是否满意？是否需要进一步调整？
```

等待用户确认后，再进行 3.5 更新计划文件。

#### 3.5 更新计划文件

追加审查结果到计划文件：
- 风险评估
- 测试策略要点
- 确定性检查结果
- 调整说明（如有）

### Phase 4: 最终计划（Final Plan）

**目标：** 整理最终计划文件

#### 4.1 整理计划文件

将计划文件整理为最终版本，结构如下：

`examples/plan-template.md`


#### 4.2 最终计划原则

- **聚焦需求和方案**，去除过程性内容
- **零上下文可执行**，假设执行者对代码库零了解，计划必须自包含所有执行所需信息
- **确定性优先**，每个任务描述必须无歧义，任何人执行都产出相同结果
- **规约完整**（复杂/系统级），数据模型、接口契约、业务规则不留模糊空间
- **便于用户审核**，快速判断需求和方案是否符合预期
- **便于AI执行**，对照文档即可准确实现需求
- **步骤依赖必须填写**，根据任务实际依赖关系填写"步骤依赖"章节，`-` 表示无依赖可并行，步骤名表示必须等待前置步骤完成

#### 4.3 执行步骤编写要求

每个执行步骤必须满足以下要求，禁止使用描述性语言代替具体内容：

**文件区域（必填）：** 明确列出涉及的文件及操作类型
```markdown
**涉及文件：**
- 新增: `exact/path/to/new-file.ts`
- 修改: `exact/path/to/existing.ts`
- 测试: `tests/exact/path/to/test.ts`
```

**代码片段（必填）：** 包含完整的、可直接使用的代码，不得用描述代替
```markdown
# 好：包含完整代码
在 `src/utils/validate.ts` 中添加：
​```typescript
export function validateEmail(email: string): boolean {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}
​```

# 坏：用描述代替代码（禁止）
"添加邮箱验证函数"
"实现验证逻辑"
```

**验证命令（必填）：** 包含精确的运行命令和预期输出
```markdown
**验证：**
运行: `npm test -- tests/utils/validate.test.ts`
预期: 所有测试通过，输出 "X tests passed"
```

## 复杂度适配

根据任务复杂度动态调整流程深度：

| 复杂度 | 特征 | 流程调整 |
|--------|------|----------|
| **简单** | 单一技术方案、低风险、单模块 | 快速通过 Phase 1-2，简化 Phase 3；跳过 WBS 工作分解、模块规约、技术性 AC |
| **中等** | 2-3个技术方案、中等风险、2-3个模块 | 标准流程；执行 WBS 工作分解；跳过模块规约、技术性 AC |
| **复杂** | 多个技术方案、高风险、多模块（4+）、涉及架构决策 | 深入每个阶段；执行 WBS 工作分解 + 模块规约 + 技术性 AC + 确定性检查 |
| **系统级** | 从零构建完整系统、涉及多端/多服务、需要数据库设计和接口设计 | 所有步骤必须执行；每个模块完整规约；AC 逐项审核 |

**复杂度判定规则：**
- 从头搭建一个新系统 → 默认系统级
- 涉及数据库设计 + API 设计 + 前端页面 → 至少复杂级
- 涉及到多个（4+）功能模块或者庞大的（可以往下拆分三层子功能的）功能模块或者开发时长超过 7 天 → 至少复杂级
- 单个文件或单个模块的修改 → 简单或中等
