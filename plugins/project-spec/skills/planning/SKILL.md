# 交互式计划制定 Skill

## 概述

此 skill 用于把模糊需求收敛成一份可审查、可执行、可交接的实施计划。

它的核心目标不是“尽快开始写代码”，而是先回答四个问题：
- 要解决什么问题
- 哪些范围必须做，哪些不做
- 推荐采用什么方案，为什么
- 任何执行者如何按计划稳定落地

默认采用 5 个阶段推进：深度理解 → 方案设计 → 审查对齐 → 最终计划 → 退出规划。

## 何时使用

在以下场景使用此 skill：
- 用户说“帮我制定计划”“把任务拆解一下”“规划一下这个需求”
- 需求还不完整，需要先澄清边界、角色、流程或约束
- 存在多个技术方案，需要比较后再决定
- 任务规模较大，直接开始实现风险高
- 需要输出 `docs/spec/` 下的计划文件，供 `/ps-exec` 或人工执行
- 需要在编码前先完成风险评估、范围确认和验收标准定义

## 何时不要使用

以下情况通常不必进入完整规划流程：
- 需求已经非常明确，且只是低风险的小改动
- 单文件、小范围修复，用户明确要求直接实现
- 当前任务重点是排查 bug，应优先使用调试类 skill
- 当前任务重点是立即编码，应优先遵循实现类 workflow

如果不确定是否要进入完整流程，先做复杂度判定。

## 核心规则

1. **规划阶段默认只读**
   - 允许读取代码、文档、配置、测试。
   - 只允许写计划文件，不修改业务代码或配置。

2. **先理解，再设计，再承诺**
   - 不要在需求和约束未清之前给出最终方案。
   - 不要在方案未收敛之前输出详细执行步骤。

3. **用户决策点必须显式确认**
   - 在 Claude Code 插件环境中，所有正式提问都使用 `AskUserQuestion`。
   - 若当前环境没有该工具，使用等价的结构化提问方式代替。
   - 每次只问一组相关问题，控制在 2-4 个独立问题内。

4. **复杂任务才强制重流程**
   - 并行子代理探索、WBS 分解、模块规约、技术性验收标准，只在中等及以上复杂度逐步引入。
   - 不要把系统级流程硬套到简单任务上。

5. **计划文件必须面向零上下文执行者**
   - 最终计划默认假设执行者对代码库不熟悉。
   - 执行步骤必须包含精确路径、关键代码片段、验证命令和预期结果。

6. **先删范围，再加细节**
   - 先确认必须做什么，再细化如何做。
   - 任何“可有可无”的功能都先经过 YAGNI 检查。

## 复杂度适配

| 复杂度 | 典型特征 | 必做步骤 | 可跳过步骤 |
|--------|----------|----------|------------|
| 简单 | 单模块、小范围、低风险、方案明确 | Phase 1、Phase 2、简化 Phase 3 | WBS、模块规约、技术性 AC |
| 中等 | 2-3 个模块、中等风险、需要拆解 | WBS、范围确认、风险检查 | 模块规约、技术性 AC |
| 复杂 | 多模块、高风险、涉及架构或接口设计 | 全流程；并行子代理探索；WBS；模块规约；技术性 AC | 无 |
| 系统级 | 从零构建系统，多服务/多端/数据库/API 联动 | 全流程；每个模块都要完整规约 | 无 |

**默认判定规则：**
- 从零搭建一个系统，默认按“系统级”处理
- 涉及数据库设计、接口设计、前端页面三者中的两项及以上，至少按“复杂”处理
- 涉及 4 个及以上功能模块，或预计开发超过 7 天，至少按“复杂”处理
- 仅单文件或单模块修改，通常是“简单”或“中等”

## 工作流

### Phase 1：深度理解

**目标：** 搞清楚需求、现状、约束和待决策问题。

#### 1. 先判定复杂度

先根据任务规模、风险和模块数量判断是简单、中等、复杂还是系统级。  
这一步决定后续是否必须使用并行子代理、WBS 分解和模块规约。

#### 2. 收集上下文

至少完成以下内容：
- 阅读相关命令、文档、模板和关键代码
- 找出现有实现模式、可复用模块和约束
- 识别关键文件、依赖关系和验证方式

#### 3. 按复杂度决定是否并行探索

**简单任务：**
- 主 Agent 直接读取关键文件即可

**中等任务：**
- 如信息分散，可按 2-3 个维度启动并行探索子代理

**复杂/系统级任务：**
- 必须先做维度分析，再为每个维度启动独立探索子代理并行探索
- 不要把多个不相关维度塞进同一个子代理

推荐探索维度：
- 现有实现模式
- 相关模块与依赖
- 测试与验证方式
- 文档、规范与外部约束

#### 4. 用 5 维度框架梳理需求

以下维度用于内部思考，不必原样抛给用户：
- **目标与背景**：为什么做，成功标准是什么
- **用户与角色**：谁使用，权限或角色差异是什么
- **核心流程与交互**：主路径、状态流转、分支路径是什么
- **数据与内容**：涉及哪些实体、字段、关系、展示内容
- **边界与规则**：限制条件、异常场景、明确不做什么

#### 5. 中等及以上任务做 WBS 分解

将任务拆成 `Epic → Feature → Story` 三层。

**Story 叶节点标准：**
- 工作量约 2-6 小时
- 涉及文件数不超过 3 个
- 新增或修改代码通常不超过 200 行
- 测试点通常不超过 5 个
- 外部依赖不超过 2 个
- 不再依赖用户做关键决策
- 可以用一句话明确描述

**继续下钻的信号：**
- Story 仍然模糊，无法直接执行
- Story 同时包含多个独立动作
- 预计超过 6 小时
- 仍依赖用户决定规则、方案或边界

#### 6. 澄清真正关键的歧义

提问时遵守：
- 优先多选题
- 每次只问一组相关问题
- 不重复询问用户已经明确说明的内容
- 可以根据常识和现有代码推断的内容，不要再问

#### 7. 输出理解摘要并确认

在写入计划文件前，先给用户一个简短摘要，至少包含：
- 任务目标
- 核心需求
- 关键约束
- 已知风险或未知项

在支持 `AskUserQuestion` 的环境中，摘要展示后立即通过该工具收集确认；否则使用等价的结构化确认方式。  
确认无误后，再进入 Phase 1 的计划文件更新。

#### 8. 更新计划文件

保存到：

`docs/spec/[YYYYMMDD-HHmm]-[功能名称].md`

如果目录不存在，先创建 `docs/spec` 目录。

Phase 1 结束后，计划文件至少应包含：
- 背景与目标
- 需求概览
- 探索发现
- 功能分解树（中等及以上）
- 用户故事级验收条件（复杂及以上）

### Phase 2：方案设计

**目标：** 确定推荐方案、技术边界和可执行的实现方向。

#### 1. 先收集技术约束

重点关注：
- 性能、可用性、安全性、可扩展性
- 技术栈、版本、预算、时间、团队约束
- 现有系统、数据库、API、第三方服务

如果 Phase 1 已经确认部分技术约束，直接复用，不重复提问。

#### 2. 设计候选方案并做取舍

存在多个方案时，至少从以下维度比较：
- 实现复杂度
- 性能
- 可维护性
- 兼容性
- 安全性
- 成本

必须给出：
- 候选方案
- 各自优缺点
- 推荐方案
- 推荐理由

#### 3. 复杂/系统级任务补齐模块规约

对每个一级模块产出技术契约，至少包含：
- 数据模型：字段、类型、约束、关系
- 接口契约：路径、方法、入参、出参、状态码、错误码
- 业务规则：使用明确的 `if / then / else` 表达
- 页面或组件规约：涉及前端时说明组件、状态和交互
- 关键决策记录：为什么选这个方案

如果环境支持规划类子代理，且模块之间可独立分析，可并行生成规约。

#### 4. 复杂/系统级任务补齐技术性验收标准

使用 `Given / When / Then` 编写。  
必须覆盖正常路径和异常路径。

异常路径至少考虑以下维度中的 2 个：
- 数据异常
- 网络异常
- 权限异常
- 并发异常

#### 5. 做一次范围再确认

此时已经知道大概工作量，要主动删除性价比低的功能。

至少明确：
- 保留什么
- 删除什么
- 降级什么
- 为什么这样裁剪

#### 6. 输出方案摘要并确认

摘要至少包含：
- 推荐方案
- 核心步骤
- 范围取舍
- 关键风险与缓解措施

在支持 `AskUserQuestion` 的环境中，摘要展示后立即通过该工具收集确认；否则使用等价的结构化确认方式。  
确认后再更新计划文件。

#### 7. 更新计划文件

Phase 2 结束后，计划文件至少补充：
- 方案概述
- 方案对比（如有）
- 模块规约（复杂/系统级）
- 技术性验收标准（复杂/系统级）
- 初版执行步骤

### Phase 3：审查对齐

**目标：** 验证方案是否完整、可执行、与用户意图一致。

#### 1. 回读关键文件

重新核对：
- 关键代码和文档是否支持当前方案
- 计划里的路径、依赖、命名是否真实存在
- 有没有遗漏已经存在的实现模式

#### 2. 做风险评估

至少评估以下风险：
- 技术风险
- 依赖风险
- 性能风险
- 安全风险
- 兼容性风险
- 时间风险

每个高优先级风险都要给出缓解措施或备选方案。

#### 3. 做完整性和确定性检查

逐项检查：
- 所有需求是否都有对应执行步骤
- 所有高风险点是否都有缓解措施
- 数据模型、接口契约、业务规则是否有遗漏
- 文件路径、依赖关系、验证命令是否真实可用
- 是否存在“适当的”“合理处理”“根据情况”等模糊词

如果发现不确定项：
- 能通过代码或文档确认的，立即确认并更新计划
- 必须由用户拍板的，集中提问
- 属于惯例实现的，采用惯例并写入计划

#### 4. 澄清剩余问题

继续使用结构化提问，但只问真正影响方案正确性的剩余问题。  
不要把审查阶段变成第二轮发散式需求讨论。

#### 5. 输出审查摘要并确认

至少包含：
- 主要风险和缓解措施
- 测试策略要点
- 发现并修正的不一致项
- 剩余待确认问题

在支持 `AskUserQuestion` 的环境中，摘要展示后立即通过该工具收集确认；否则使用等价的结构化确认方式。  
确认后再更新计划文件。

#### 6. 更新计划文件

Phase 3 结束后，计划文件至少补充：
- 风险评估
- 测试策略
- 完整性与确定性检查结果
- 调整说明

### Phase 4：最终计划

**目标：** 把分散信息整理成最终可执行版本。

#### 1. 使用模板整理

按以下模板整理同一份计划文件，逐阶段收敛为最终版：

`examples/plan-template.md`

如果后续要交给 `/ps-exec` 消费，最终章节结构还必须与执行器的解析约定保持一致；模板与执行器不一致时，以执行器要求为准补齐缺失章节。

#### 2. 最终计划必须满足的要求

- **聚焦需求和方案**：删掉过程性对话内容
- **零上下文可执行**：任何执行者都能直接照做
- **确定性优先**：避免模糊词和开放式任务描述
- **依赖明确**：每个步骤都写清前置依赖
- **便于审核**：用户能快速判断范围和方案是否正确
- **便于执行**：AI 或人工可以直接按步骤落地

#### 3. 执行步骤写法

每个步骤至少包含：
- **目标**
- **涉及文件**
- **操作说明**
- **关键代码片段**
- **验证命令**
- **预期结果**
- **步骤依赖**

对需要直接落地的代码变更，不要只写“添加逻辑”“实现功能”，要给出足以执行的具体内容。

### Phase 5：退出规划

**目标：** 结束规划流程，并为后续执行留出清晰入口。

#### 1. 展示最终摘要

至少概括：
- 解决的问题
- 推荐方案
- 核心步骤数量
- 关键风险
- 已确认与未确认事项

#### 2. 请求用户批准

确认用户是否接受当前计划文件作为执行基线。

#### 3. 询问是否进入执行

如果当前环境支持命令工作流，使用 `AskUserQuestion` 或等价交互方式询问用户是否继续执行：
- 立即执行：使用 `/ps-exec [计划文件路径]`
- 稍后执行：保留计划文件，等待后续执行

使用 Skill tool 加载 ps-exec 执行计划

## 计划文件增量更新规则

| 阶段 | 必须写入的内容 |
|------|----------------|
| Phase 1 完成后 | 背景与目标、需求概览、探索发现、功能分解树、用户故事级验收条件 |
| Phase 2 完成后 | 方案概述、方案对比、模块规约、技术性验收标准、初版执行步骤 |
| Phase 3 完成后 | 风险评估、测试策略、完整性检查结果、调整说明 |
| Phase 4 完成后 | 最终版整理完成，可直接交付执行 |

## 常见错误

### 1. 还没理解需求就急着设计方案

表现：
- 用户目标、范围和边界还没确认，就直接给技术选型

修正：
- 先完成 Phase 1，再进入 Phase 2

### 2. 简单任务也强行走系统级流程

表现：
- 单文件改动也启动多个 Agent、输出完整 WBS 和模块规约

修正：
- 先做复杂度判定，按复杂度裁剪流程

### 3. 提问过多、过散、过早

表现：
- 一次抛出十几个问题
- 用户已经说过的信息还在重复问

修正：
- 每次只问一组关键问题
- 能从代码和上下文推断的内容先自己确认

### 4. 计划步骤不可执行

表现：
- 只写“实现认证”“增加校验”
- 不写文件路径、代码片段、验证命令

修正：
- 执行步骤必须写成可直接落地的操作说明

### 5. 计划和实际代码库脱节

表现：
- 路径不存在
- 引用的模块或脚本名不正确

修正：
- 在 Phase 3 回读关键文件并做一致性检查

### 6. 范围只增不减

表现：
- 所有想到的功能都放进计划，没有取舍

修正：
- 在 Phase 2 和 Phase 3 都做 YAGNI 检查

## 最终检查清单

完成规划前，至少确认以下事项：

- [ ] 已完成复杂度判定，并按复杂度裁剪流程
- [ ] 已收集相关代码、文档、依赖和验证方式
- [ ] 关键歧义已经澄清，未澄清项已明确标出
- [ ] 推荐方案、备选方案和取舍理由清楚
- [ ] 中等及以上任务已完成合理粒度的功能拆解
- [ ] 复杂/系统级任务已补齐模块规约和技术性验收标准
- [ ] 风险、测试策略、范围取舍都已写入计划
- [ ] 最终计划使用模板整理，执行步骤可直接落地
- [ ] 计划文件路径正确：`docs/spec/[YYYYMMDD-HHmm]-[功能名称].md`
- [ ] 用户已明确确认是否继续进入 `/ps-exec`
